# XZ Update Plan - 修改版本计划的 todolist

你收到的参数: `$ARGUMENTS`

其中 `$0` 是版本号 N，`$0` 之后的内容是操作描述（修改/新增/删除要求）。

### 参数校验

如果 `$ARGUMENTS` 为空或 `$0` 不是正整数，**立即停止**，提示：

> 缺少版本号。用法: `/xz-update-plan N 操作描述`
> 示例: `/xz-update-plan 1 修改 #3 增加 refresh token 逻辑`

**示例调用：**
- `/xz-update-plan 1 修改 #3 增加 refresh token 逻辑`
- `/xz-update-plan 1 新增一条: 添加日志中间件`
- `/xz-update-plan 1 删除 #4`
- `/xz-update-plan 1 在 #2 后面插入一条缓存优化`

---

## 辅助脚本

**脚本路径**：取本 skill 的 Base directory（prompt 开头 "Base directory for this skill:" 的值），向上两级目录，拼接 `bin/xz-tools.py`。

示例：Base directory = `.../skills/xz-update-plan` → 脚本 = `.../bin/xz-tools.py`

后续所有调用使用 `python3 <脚本绝对路径> <命令>` 格式。脚本在**当前工作目录**下操作 `.xz_planning/`。

## 执行流程

### 第一步：读取项目上下文

读取 `.xz_planning/PROJECT.md`（如果存在），快速了解项目技术栈、目录结构和关键入口，为后续新增/修改条目的 change details 提供准确的文件路径参考。

### 第二步：解析当前计划

```bash
python3 <脚本绝对路径> parse $0
```

- 如果返回 `"ok": false` → 版本不存在，提示用户先运行 `/xz-plan N 需求描述` 创建计划
- 如果返回 `"ok": true` → 读取 N-PLAN.md 完整内容，继续

### 第三步：读取并展示当前 todolist

读取 N-PLAN.md 文件，展示当前所有条目及状态：

```
当前 todolist (版本 N):
  [x] #1 创建 User model           ← 已完成，锁定
  [x] #2 实现注册接口               ← 已完成，锁定
  [ ] #3 实现登录接口               ← 可修改
  [ ] #4 JWT 中间件鉴权             ← 可修改
  [ ] #5 添加单元测试               ← 可修改
```

### 第四步：锁定检查

**铁律：已完成的 `[x]` 条目不可修改、不可删除。**

如果用户的操作描述涉及 `[x]` 条目，**立即拒绝**并说明：

> #N 已完成，无法修改/删除。已完成的条目受锁定保护。

### 第五步：解析用户操作意图

根据操作描述，判断属于以下哪种操作（可以组合）：

#### 操作 A：修改已有条目

- 用户指定编号（如 "修改 #3"）
- 检查该编号是否为 `[ ]` 状态
- 生成修改后的条目内容（标题 + change details）

#### 操作 B：新增条目

- 在末尾追加，或在指定位置插入（如 "在 #2 后插入"）
- 插入后，后续条目编号自动递增
- 新条目必须包含完整的 change details

#### 操作 C：删除条目

- 用户指定编号（如 "删除 #4"）
- 检查该编号是否为 `[ ]` 状态
- 删除后，后续条目编号自动递减

### 第六步：方案对比（如有多种改法）

与 `/xz-plan` 同理，在生成变更草案前**必须先判断**：用户描述的修改/新增是否存在多种实现方式？

**判断条件（任一命中就出方案对比）：**

1. 修改涉及技术选型变更（换库、换协议、换数据结构）？
2. 新增条目的实现路径不止一种？
3. 修改会影响其他已有条目的 change details？
4. 存在性能 vs 复杂度 vs 兼容性的权衡？

**命中 → 出方案对比：**

```markdown
当前修改存在多种实现方式：

### 方案 A (推荐): 方案名称

**核心思路:** 一句话说清楚

**涉及改动:**
- 修改 #3: xxx → xxx
- 新增 #6: xxx

**优点:** ...
**缺点:** ...

---

### 方案 B (备选): 方案名称

**涉及改动:**
- 修改 #3: xxx → xxx
- 修改 #4: xxx → xxx（连带影响）
- 新增 #6: xxx

**优点:** ...
**缺点:** ...

---

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 改动条目数 | 2 | 3 |
| 复杂度 | 低 | 中 |
| 对已有条目影响 | 无 | #4 需连带修改 |

👉 推荐方案 A: [理由]
```

**输出方案对比后，使用 AskUserQuestion 工具让用户选择方案，禁止自动选方案：**

- question: "选择实现方案："
- header: "方案选择"
- options: 动态根据实际方案数构建，每个选项的 label 为方案名（如 "方案 A (推荐): xxx"），description 为核心思路一句话摘要
- multiSelect: false

用户选定后，按选定方案进入下一步。如果用户选择 Other，按其输入的方案偏好处理。

**未命中 → 直接进入下一步。** 简单的增删改（如"删除 #4"、"修改 #3 的描述文字"）不需要出方案。

### 第七步：生成变更草案

**先输出到对话中，不写文件！**

展示变更对比，格式：

```
变更对比 (版本 N):

  [锁定] #1 创建 User model         ✅ 已完成，不可变更
  [锁定] #2 实现注册接口             ✅ 已完成，不可变更
  [修改] #3 实现登录接口
    旧: JWT 签发，过期 24h
    新: JWT 签发 + Refresh Token，access 过期 1h，refresh 过期 7d
  [保留] #4 JWT 中间件鉴权           （无变化）
  [删除] #5 添加单元测试             ← 将被删除
  [新增] #5 添加 token 刷新接口      ← 新增条目

重编号后最终 todolist:
  [x] #1 创建 User model
  [x] #2 实现注册接口
  [ ] #3 实现登录接口 (已修改)
  [ ] #4 JWT 中间件鉴权
  [ ] #5 添加 token 刷新接口 (新增)
```

### 第八步：等待确认

使用 AskUserQuestion 工具确认变更：

- question: "以上变更是否确认？"
- header: "确认变更"
- options:
  - label: "确认更新", description: "写入 N-PLAN.md 并刷新 STATE.md"
  - label: "修改意见", description: "调整后重新输出变更草案"
- multiSelect: false

**禁止自动写入文件。必须等待明确确认。** 用户选择「修改意见」或 Other 时，按其反馈调整后重新输出草案。

### 第九步：确认后更新文件

**获取当前时间：**

```bash
date "+%Y-%m-%d %H:%M:%S"
```

1. **更新 N-PLAN.md**：
   - 修改/新增/删除对应的 todolist 条目和 change details
   - 重新编号所有条目，保证连续（1, 2, 3, ...）
   - **同步更新「任务总结」章节** — 根据变更后的 todolist 重新生成自然语言总结，保持与条目一一对应
   - 更新「最后更新」时间
   - 在「变更记录」中追加一行：`- YYYY-MM-DD HH:mm:ss 操作描述`

2. **刷新 STATE.md**：

```bash
python3 <脚本绝对路径> update-state
```

### 第十步：输出改动结果

整理并输出：
- 修改了哪些条目
- 新增了哪些条目
- 删除了哪些条目
- 当前 todolist 总数和完成进度

然后使用 AskUserQuestion 工具让用户选择下一步操作：

- question: "计划已更新。接下来要做什么？"
- header: "下一步"
- options:
  - label: "/xz-exec N", description: "继续执行 todolist"
  - label: "/xz-update-plan N", description: "继续修改计划"
  - label: "/xz-ref N", description: "加载计划作为上下文参考"
- multiSelect: false

用户选择后，执行对应的 skill 命令。如果用户选择 Other，按其输入内容响应。

## 关键规则

1. **已完成锁定** — `[x]` 条目绝不可修改或删除，这是铁律
2. **先输出后写文件** — 变更草案必须先展示，用户确认后才写入
3. **编号自动维护** — 插入/删除后重新编号，保持 1, 2, 3... 连续
4. **change details 必填** — 新增/修改的条目必须包含完整的 change details
5. **变更记录追加** — 每次修改都在变更记录中留痕
6. **STATE.md 同步** — 更新文件后必须刷新 STATE.md 反映最新进度
7. **多种改法必须出方案** — 涉及技术选型变更、实现路径不唯一、或影响其他条目时，必须先出 A/B 方案对比，等用户选定后再生成变更草案。简单增删改不需要
8. **Bug/问题排查类变更** — 如果操作描述涉及修复 bug、排查问题、处理异常等，建议用户先用 `/xz-debug` 定位根因，再回来新增/修改 todolist 条目。纯功能调整不需要
