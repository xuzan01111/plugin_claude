# Bug 排查与修复

根据用户描述的问题现象，系统性地查找 bug 并给出修复方案。

## 排查流程

### 1. 理解问题

从 `$ARGUMENTS` 中提取：
- 问题现象（报错信息、异常行为、预期 vs 实际）
- 涉及的功能模块或文件（如有提及）
- 复现条件（如有提及）

如果描述不够清晰，输出以下内容后结束响应，等用户补充：
- 目前已理解的部分
- 还需要的具体信息（如：完整报错信息、复现步骤、触发时的输入数据、运行环境）

### 2. 定位问题

按以下优先级逐步排查：

1. **错误信息追踪**：如果有报错信息，从错误堆栈或关键字出发，用 Grep 搜索相关代码
2. **入口定位**：根据功能描述，找到对应的路由、页面、组件或 API 入口
3. **调用链分析**：沿调用链逐层深入，检查数据流和控制流
4. **关联影响**：检查相关模块是否受影响

**排查中遇到无法仅靠代码确认的情况时：**

立即停止猜测，输出以下内容后结束响应，等用户提供：

1. **已排查的范围** — 读了哪些文件、检查了哪些调用链
2. **当前发现** — 找到的可疑点，附代码位置
3. **无法确认的原因** — 为什么靠静态代码无法定下结论（如：需要运行时日志、需要知道实际传入的参数值、需要确认环境配置、需要数据库中的实际数据）
4. **需要用户提供的具体信息** — 明确列出，每条一行，让用户可以逐条回复

示例：

> 已排查 `src/auth/login.py:45-82` 的登录流程和 `src/db/user.py:12-30` 的查询逻辑。
> 发现两个可疑点：
> 1. `login.py:67` — token 过期判断用的是 `<=` 而非 `<`，边界条件可能不对
> 2. `user.py:23` — 查询没有处理用户不存在的情况，会返回 None
>
> 但无法确认哪个是实际触发的原因，需要你提供：
> - 报错时的完整日志输出（含时间戳）
> - 触发时使用的用户账号（确认是否存在于数据库）
> - token 的签发时间和过期时间

### 3. 列出问题点

对每个发现的问题，按以下格式输出：

```
问题 N：<简要标题>
- 位置：<文件路径:行号>
- 现象：<该问题导致的具体表现>
- 原因：<根本原因分析>
- 严重程度：严重 / 高 / 中 / 低
- 确信度：确认 / 高度疑似 / 待验证
```

**确信度说明：**
- **确认** — 代码逻辑明确有 bug（如类型错误、空指针、逻辑反了）
- **高度疑似** — 代码有问题但需要特定运行时条件才触发，已标注触发条件
- **待验证** — 代码看起来可疑但无法仅靠静态分析确认，已列出需要用户提供什么信息来验证

按严重程度从高到低排列。

### 4. 给出解决方式

对每个问题点，给出：
- **修复方案**：具体的代码修改建议，包含修改前后对比
- **注意事项**：修复时需要关注的边界条件、兼容性或副作用
- **验证方式**：如何确认修复有效

### 5. 总结

- 问题数量和严重程度分布
- 根本原因归类（逻辑错误、类型错误、竞态条件、边界处理等）
- 是否建议进一步检查其他相关模块

## 排查原则

- 优先查找最可能导致问题的代码，不要漫无目的地搜索
- 区分"根本原因"和"表面现象"，修复要针对根因
- 如果存在多个问题，判断是否有因果关系，先修复上游问题
- 不要猜测，所有结论必须基于实际代码
- 信息不足时停下来反问，不模糊推测 — 把已有发现和需要的信息列清楚，让用户补充后再继续
- 多个疑点无法区分时全部列出并标注确信度 — 不要随意挑一个当结论，让用户结合运行时信息判断
