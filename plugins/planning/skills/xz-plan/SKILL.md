# XZ Plan - 创建新版本计划

你收到的参数: `$ARGUMENTS`

**参数解析规则：** 第一个空格前的内容为版本号 N，第一个空格后的所有内容为需求描述。
示例: `$ARGUMENTS` = `2 支持多语言切换` → 版本号=2，需求描述="支持多语言切换"

### 参数校验

1. 如果 `$ARGUMENTS` 为空，**立即停止**，提示：

> 缺少参数。用法: `/xz-plan N 需求描述`
> 示例: `/xz-plan 1 支持多语言切换`

2. 如果版本号不是正整数，**立即停止**，同上提示。

3. 如果需求描述为空（只传了版本号），**立即停止**，提示：

> 缺少需求描述。用法: `/xz-plan N 需求描述`
> 示例: `/xz-plan 1 支持多语言切换`

---

## 辅助脚本

插件 `bin/` 目录下的 `xz-tools.py`（插件启用时自动加入 PATH）

后续所有脚本调用直接使用 `xz-tools.py`。脚本在**当前工作目录**下操作 `.xz_planning/`。

---

## 执行流程

### 第一步：检查是否已初始化

检查当前目录下 `.xz_planning/` 是否存在。

- **不存在** → **停止！** 提示用户：

> 项目尚未初始化。请先执行 `/xz-init` 创建计划目录。

- **已存在** → 继续

### 第二步：检查版本是否已存在

```bash
xz-tools.py parse N
```

（将 N 替换为用户传入的实际版本号）

| 返回结果 | 含义 | 操作 |
|----------|------|------|
| `"ok": false` 且 `phase` 为空 | 版本不存在 | 继续新建流程 |
| `"ok": false` 但 `phase` 存在 | 有目录但无 PLAN（如只有 DISCUSS） | 继续新建流程，目录复用 |
| `"ok": true` | PLAN 已存在 | **停止！** 提示用户使用 `/xz-update-plan N` 修改 |

## 新建计划

### 1. 分析需求 + 代码现状

**必须**先做以下调研，不可跳过：

1. **读取项目快照** — 先读 `.xz_planning/PROJECT.md`（如果存在），快速了解技术栈、目录结构、关键入口
2. **读取讨论文档** — 检查同目录下是否存在 `N-DISCUSS.md`，如果存在则读取完整内容，将其中的目标用户、功能方向、MVP 收敛、技术边界等作为规划的重要输入参考
3. **阅读当前项目代码** — 在 PROJECT.md 基础上，用 Glob/Read 深入了解需求涉及的具体模块和文件
4. **理解需求描述** — 明确用户到底要做什么（如有 DISCUSS 文档，优先参考其中已收敛的结论）
5. **识别影响范围** — 需要新建哪些文件、修改哪些现有文件、依赖什么

**扫描排除规则：**
- 排除 `.xz_planning/` 目录
- 排除 `.gitignore` 中声明的文件和目录
- 只关注项目实际源码和配置文件

### 2. 范围检查

如果需求涉及**多个独立子系统**（彼此没有依赖关系），**发现需拆分后立即停止，输出拆分建议，不继续分析子系统细节**。建议拆分为多个版本，每个子系统独立走 `/xz-plan`。每个计划应该能独立产出可工作、可验证的结果。

**不包含测试计划** — todolist 只规划功能实现，测试由 `/xz-test N` 独立负责。

**Bug/问题排查类需求（强制触发 xz-debug）：**
如果需求描述属于以下任一情况，**必须先调用 xz-debug skill 完成排查，拿到确认的问题点和修复方案后，再回到 `/xz-plan` 规划修复 todolist**：
- 用户描述的是 bug、报错、异常行为、逻辑错误
- 需求本质是"修复"而非"新增功能"（如"登录失败""数据不对""页面白屏"）
- 用户提供了错误信息、堆栈、截图等问题现象

**流程：** 检测到上述情况 → 输出提示"`该需求属于问题排查，将先通过 xz-debug 定位根因`"后结束响应 → 等用户确认后执行 xz-debug → 拿到排查结果后再继续 `/xz-plan` 生成修复 todolist。

**禁止：** 跳过排查直接猜测原因并规划修复方案。没有经过 xz-debug 确认的问题点不得写入 todolist。

### 3. 方案评估

基于第 1 步的代码调研结果，判断实现方式有几种：

**只有一种做法** → 直接说明这条方案的思路，进入第 4 步生成 todolist。

**有多种做法** → 列出 A/B/C 方案对比，等用户选定后再进入第 4 步。

**铁律：**
- 方案必须来自代码分析，禁止凭空编造方案凑数
- 如果只有一种合理做法，就一条，不要为了"看起来专业"硬凑多条
- 如果有多条，每条之间必须有实质差异（不同的库、不同的架构、不同的改法），不是措辞不同
- 列完方案后**使用 AskUserQuestion 工具让用户选择**，禁止自动选方案

**多方案对比格式：**

每个方案必须包含以下内容：

```markdown
### 方案 A (推荐): 方案名称

**核心思路:** 一句话说清楚这个方案怎么做

**涉及改动:**
- 新建: xxx — 做什么
- 修改: xxx — 改什么
- 依赖: xxx — 为什么需要

**优点:**
- ...

**缺点:**
- ...

**适用场景:** 什么情况下选这个最合适

---

### 方案 B (备选): 方案名称

（同上格式）
```

**优缺点必须具体量化，禁止空泛表述：**

错误: "性能好"、"开发快"、"扩展性强"
正确: "首次加载减少约 40KB（支持 tree-shaking）"
正确: "比方案 B 少改 2 个文件，不动现有 API 接口"
正确: "需额外维护映射表，每次加新语言要改 2 处"
正确: "方案 A 改 3 个文件约 45 行，方案 B 改 1 个文件约 80 行"

**对比总结表（必须附上）：**

```markdown
| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 改动范围 | 3 文件 | 5 文件 | 1 文件 |
| 开发量 | 中 | 长 | 短 |
| 性能 | 高 | 最高 | 一般 |
| 可维护性 | 好 | 复杂 | 简单 |
| 扩展性 | 好 | 最好 | 差 |

推荐方案 A: [推荐理由]
```

**决策：** 输出方案对比后，使用 AskUserQuestion 工具让用户选择方案，禁止自动选择方案，禁止跳过直接生成 todolist：

- question: "选择实现方案："
- header: "方案选择"
- options: 动态根据实际方案数构建，每个选项的 label 为方案名（如 "方案 A (推荐): xxx"），description 为核心思路一句话摘要
- multiSelect: false

用户选定后，再根据该方案进入第 4 步生成 todolist。如果用户选择 Other，按其输入的方案偏好处理。

### 4. 生成 todolist 草案

**先输出到对话中，不写文件！**

**获取当前时间：**

```bash
date "+%Y-%m-%d %H:%M:%S"
```

**change details 行号规则（必须遵守）：**

1. **修改文件必须标注行号范围** — 格式: `文件路径:起始行-结束行`
2. **行号旁标注当前内容摘要** — 用括号写明这几行现在是什么，格式: `(现: 简述当前内容)`
3. **新建文件不需要行号** — 但要列出将包含的函数/类/常量名称
4. **行号在生成 todolist 前通过 Read 工具确认** — 不得凭记忆猜测行号
5. **如果修改区域超过 30 行，拆成多个子区域分别标注**

**粒度要求：**
- 每条 todo 是一个 2-5 分钟可完成的原子操作
- 能独立完成、独立验证
- 一条 todo 不要跨太多文件，一个功能单元一条

**格式模板：**

````markdown
# 版本 N: 自动生成的简短中文需求名,最多 8 个中文字

> 创建时间: YYYY-MM-DD HH:mm:ss
> 最后更新: YYYY-MM-DD HH:mm:ss
> 状态: 待执行

## 需求描述
用户输入的原始需求完整描述。

## 技术方案
（如果有多方案对比，记录选定的方案及理由；只有一种做法时简述实现思路）

## 文件变更清单

**新建:**
- `path/to/new-file` — 职责说明（一句话）

**修改:**
- `path/to/existing:行号范围` — 改什么（一句话）

**依赖:**
- 包名 — 用途

## Todolist

- [ ] 1. 创建多语言配置文件

  **Files:**
  - Create: `src/config/i18n.ts`

  ```
  change details:
  新建: src/config/i18n.ts
  - 导出 supportedLocales: string[] = ["zh-CN", "en-US", "ja-JP"]
  - 导出 defaultLocale = "zh-CN"
  - 导出 loadMessages(locale: string) 函数，从 locales/ 目录动态 import
  ```
----------------------
- [ ] 2. 创建语言包文件

  **Files:**
  - Create: `src/locales/zh-CN.json`
  - Create: `src/locales/en-US.json`

  ```
  change details:
  新建: src/locales/zh-CN.json
  - 包含所有界面文本的中文翻译键值对
  - 按页面/模块分组（common, home, settings）
  新建: src/locales/en-US.json
  - 同上结构的英文翻译
  ```
----------------------
- [ ] 3. 集成语言切换到界面

  **Files:**
  - Modify: `src/components/Header.tsx:42-58`
  - Modify: `src/app.tsx:1-5, 12-18`

  ```
  change details:
  修改: src/components/Header.tsx:42-58 (现: NavRight 组件只有用户头像, 改后: NavRight 组件头像限制去掉了)
  - 在 <Avatar> 前插入 <LocaleSwitcher> 组件
  - 下拉选项来自 supportedLocales，点击调用 switchLocale()

  修改: src/app.tsx:1-5 (现: 无 i18n import，改后: 新增了 xxx)
  - 新增 import { I18nProvider } from './config/i18n'

  修改: src/app.tsx:12-18 (现: <App> 直接返回 <Router>, 改后: xxxx 变化了)
  - 外层包裹 <I18nProvider>，初始化读取 localStorage 语言偏好
  ```

## 总结

共 N 条任务，涉及 X 个文件（新建 Y 个，修改 Z 个），预计改动约 M 行。

1. 任务标题 — 自然语言说明做什么、涉及哪些文件、关键函数/组件、改成什么样
2. 任务标题 — 同上，用文字把改动流程讲清楚
3. ...

## 变更记录
- YYYY-MM-DD HH:mm:ss 初始创建
````

### 5. 等待确认

输出草案后，**必须**等待开发者回复确认。禁止自动写入文件。使用 AskUserQuestion 工具确认：

- question: "以上计划是否确认？"
- header: "计划确认"
- options:
  - label: "确认写入", description: "保存 N-PLAN.md 并刷新 STATE.md"
  - label: "修改意见", description: "调整后重新输出草案"
- multiSelect: false

用户选择「修改意见」或 Other 时，按其反馈调整后重新输出草案。

### 6. 确认后写入文件

- 如果 `.xz_planning/phases/N.xxx/` 已存在（由 `/xz-discuss` 创建） → 直接在该目录写入 `N-PLAN.md`
- 如果不存在 → 从需求描述中提取简短中文目录名（5字以内），创建目录

写入：
- 目录: `.xz_planning/phases/N.中文需求名/`
- 文件: `.xz_planning/phases/N.中文需求名/N-PLAN.md`

然后运行脚本更新 STATE.md：

```bash
xz-tools.py update-state
```

### 7. 输出结果

写入完成后，整理并输出本次操作的改动点。

输出时将 N 替换为用户传入的实际版本号：

```
计划已写入:
  .xz_planning/phases/N.xxx/N-PLAN.md
```

然后使用 AskUserQuestion 工具让用户选择下一步操作：

- question: "接下来要做什么？"
- header: "下一步"
- options:
  - label: "/xz-exec N", description: "开始执行 todolist"
  - label: "/xz-update-plan N", description: "修改/新增/删除 todolist 条目"
  - label: "/xz-ref N", description: "加载计划作为上下文参考"
- multiSelect: false

其中 N 替换为用户传入的实际版本号。用户选择后，执行对应的 skill 命令。如果用户选择 Other，按其输入内容响应。

## 关键规则

1. **先输出后写文件** — 生成的内容必须先展示，用户确认后才写入
2. **change details 必填且带行号** — 修改文件必须标注行号范围和当前内容摘要，新建文件必须列出函数/类/常量名称
3. **零上下文假设** — change details 写得像给完全不了解项目的人看，包含足够的上下文让执行者能独立理解
4. **原子化粒度** — 每条 todo 是 2-5 分钟可完成的最小操作单元，能独立完成、独立验证
5. **文件结构先行** — 文件变更清单在 todolist 草案中统一呈现，指导任务拆分
6. **中文目录自动生成** — 从需求中提取关键词作目录名，简短精炼
7. **方案来自代码分析** — 禁止凭空编造方案凑数，一种做法就一条，多种做法才列对比并等用户选
8. **调研先行** — 必须先读项目代码，不能凭空规划
9. **范围检查** — 多个独立子系统建议拆分为多个版本
10. **遵循现有模式** — 在现有代码库中遵循已有的代码风格和组织方式
11. **DRY / YAGNI** — 不做多余的设计，不为假想的未来需求预留
12. **版本已存在则拒绝** — 提示用户使用 `/xz-update-plan N` 修改
