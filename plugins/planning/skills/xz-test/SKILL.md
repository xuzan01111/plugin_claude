## 辅助脚本

**脚本路径**：取本 skill 的 Base directory（prompt 开头 "Base directory for this skill:" 的值），向上两级目录，拼接 `bin/xz-tools.py`。

示例：Base directory = `.../skills/xz-test` → 脚本 = `.../bin/xz-tools.py`

后续所有调用使用 `python3 <脚本绝对路径> <命令>` 格式。脚本在**当前工作目录**下操作 `.xz_planning/`。

## 执行流程

### 第一步：解析计划

```bash
python3 <脚本绝对路径> parse $ARGUMENTS
```

如果版本不存在，提示用户先创建计划。用返回 JSON 中的 `phase.plan_file` 获取 PLAN 文件绝对路径，UAT 文件写在同级目录。

### 第二步：扫描项目测试上下文

**了解项目怎么跑测试，为手动步骤提供准确的命令和环境信息。**

1. **查找已有测试文件**

   用 Glob 搜索：
   - `**/test_*.py`, `**/*_test.py`, `**/tests/*.py`
   - `**/*.test.ts`, `**/*.spec.ts`, `**/__tests__/*`
   - `**/test_*.go`, `**/*_test.go`
   - `**/tests/**`, `**/spec/**`

2. **识别测试框架和运行方式**

   如果找到测试文件，读取 1-2 个代表性文件，提取：
   - **测试框架**：pytest / unittest / jest / vitest / go test / 其他
   - **运行命令**：`pytest -v` / `npm test` / `go test ./...` / 其他
   - **辅助工具**：fixtures / mock / factory / 测试数据库配置

3. **检查测试配置**

   查找：`pytest.ini`, `pyproject.toml`, `setup.cfg`, `jest.config.*`, `vitest.config.*`, `Makefile`（含 test target）

4. **记录上下文摘要**（内部参考，不输出文件）

   ```
   test_context:
     framework: pytest
     run_command: pytest -v
     fixtures: tests/conftest.py (client, db_session)
     existing_tests: 8 files
   ```

   如果项目没有已有测试，记录为空，后续步骤跳过相关引用即可。

### 第三步：读取 PLAN 和代码

1. 读取 N-PLAN.md 完整内容，理解需求描述、技术方案和所有 todolist 条目
2. 读取 todolist 中涉及的**实际代码文件**，了解已实现的逻辑
3. **重点关注**：代码中是否存在明显问题（边界未处理、错误路径缺失、配置遗漏等）

### 第四步：分析问题（关键）

在读完代码后，先判断是否发现潜在问题：

- **发现问题** → 在 UAT 文件顶部用醒目的「问题发现」区块列出核心点，再给出针对性的手动验证步骤
- **未发现明显问题** → 直接生成常规手动测试场景

### 第五步：生成 N-UAT.md

在 N-PLAN.md 同级目录下创建 `N-UAT.md`：
`.xz_planning/phases/N.xxx/N-UAT.md`

**只生成这一个文件。不更新 STATE.md，不更新 PLAN.md，不生成任何测试代码文件。**

## 生成规则

1. **只生成 N-UAT.md** — 不生成测试代码文件，不生成其他任何文件
2. **问题优先展示** — 如果读代码发现了问题，必须放在文件顶部「问题发现」区块，核心点一句话讲清
3. **测试上下文参考** — 先扫描项目已有测试，利用已有框架的命令和环境信息（如运行命令、fixtures），不是凭空编造
4. **基于已完成的 `[x]` 条目生成** — 未完成的 `[ ]` 条目标注为"待实现后测试"
5. **行为描述优先** — 操作步骤用人话描述操作行为（"主播邀请观众连麦"），不是贴一堆 curl/代码。只在需要辅助验证时才给代码块
6. **前置条件要具体** — 写清楚系统状态、角色、环境、节点等上下文，让测试者知道"从什么状态开始"
7. **预期结果写正反面** — 不仅写"应该看到什么"，也写"不应该出现什么"（如"不应出现通知丢失"）
8. **覆盖正常 + 异常 + 跨模块路径** — 除了 happy path 和错误输入，还要覆盖跨节点、跨模块、并发、时序等场景
9. **每个场景对应 todolist 编号** — 标注清楚测的是哪条 todo
10. **底部附检查清单** — checkbox 列表，方便逐条打勾
11. **不更新任何状态** — 不改 PLAN.md、不改 STATE.md
