# XZ Plan - 创建新版本计划

你收到的参数: `$ARGUMENTS`

**参数解析：** 第一个空格前为版本号 N，之后全部为需求描述。示例 `2 支持多语言切换` → N=2。

### 参数校验

任一不满足 → **停止**并提示 `用法: /xz-plan N 需求描述`：
- `$ARGUMENTS` 为空
- 版本号不是正整数
- 需求描述为空

---

## 辅助脚本

**脚本路径**：取本 skill Base directory 向上两级目录拼接 `bin/xz-tools.py`。
示例：Base = `.../skills/xz-plan` → 脚本 = `.../bin/xz-tools.py`

后续调用格式：`python3 <脚本绝对路径> <命令>`。脚本在**当前工作目录**下操作 `.xz_planning/`。

---

## 执行流程

### 第一步：检查初始化

当前目录无 `.xz_planning/` → **停止**，提示 `请先执行 /xz-init`。

### 第二步：检查版本存在

```bash
python3 <脚本绝对路径> parse N
```

| 返回 | 操作 |
|------|------|
| `"ok": false` 且 `phase` 为空 | 新建 |
| `"ok": false` 但 `phase` 存在（如已有 DISCUSS） | 新建，复用目录 |
| `"ok": true` | **停止**，提示用 `/xz-update-plan N` |

---

## 新建计划

### 1. 调研需求 + 代码现状

**必做，不可跳过：**

1. 读 `.xz_planning/PROJECT.md`（若存在）了解技术栈、结构、入口
2. 读同目录 `N-DISCUSS.md`（若存在），其结论优先作为规划输入
3. 用 Glob/Read 深入需求涉及的模块
4. 识别影响范围：新建/修改文件、依赖

**扫描排除：** `.xz_planning/`、`.gitignore` 声明项。

### 2. 范围检查

- **多个独立子系统（无依赖）** → **停止**，输出拆分建议，每个子系统走独立 `/xz-plan`
- **不含测试计划** — 测试由 `/xz-test N` 负责
- **Bug/问题排查类**（报错、异常、"登录失败""数据不对"、用户贴了堆栈/截图）→ **停止**，提示 `该需求属于问题排查，先通过 xz-debug 定位根因`；**禁止凭猜测规划修复**

反例（不走 xz-debug）：性能优化、UX 改良、重构、新增功能。

### 3. 方案评估

基于第 1 步调研结果：

- **只有一种合理做法** → 简述思路，进入 3.5
- **有多种做法** → 列 A/B/C 对比，**用 AskUserQuestion 让用户选**，选定后进入 3.5

**铁律：**
- 方案来自代码分析，禁止凭空凑数；一种做法就一条
- 多方案之间必须有实质差异（不同库/架构/改法）
- 最多 3 个，超过必须合并或淘汰
- 优缺点必须量化（如 `改 3 文件约 45 行` / `首次加载减 40KB`），禁止"性能好""扩展强"

**多方案对比格式：**

```markdown
### 方案 A (推荐): 名称
**核心思路:** 一句话
**涉及改动:** 新建/修改/依赖各一句
**优点 / 缺点:** 量化表述
**适用场景:** 一句话
```

附对比总结表（维度：改动范围/开发量/性能/可维护性/扩展性）和推荐理由。

**AskUserQuestion：**
- question: "选择实现方案："
- options: 动态映射方案，label 用方案名，description 用核心思路
- multiSelect: false

### 3.5 选型澄清（一次性批量确认）

方案选定后，列出实现中**所有模糊的具体选型**（库、技术、协议、存储方式等），一次性展示。

**规则：**
- 只列真有选择空间的点（项目已固定的依赖不列）
- 每项 **a = 推荐 = 最佳**，括号一句话说优势
- 单项选项 ≤ 4 个
- 无模糊选型则跳过本步
- 用户确认后才进入第 4 步

**格式：**

```markdown
以下选型请确认（默认选 a，也是当前推荐）：

1. Web 框架
   a. Gin（性能高，社区最大，推荐）
   b. Echo（中间件生态好）
   c. Fiber（基于 fasthttp，最快但生态弱）

2. 日志库
   a. zap（结构化日志性能最佳，推荐）
   b. slog（标准库 1.21+，零依赖）

回复「全部确认」采用所有默认项，或指定如 `1b、2a` 覆盖。
```

### 4. 生成 todolist 草案

**先输出到对话，不写文件！**

获取时间：`date "+%Y-%m-%d %H:%M:%S"`

**change details 规则：**

1. 修改文件标 `path:起始行-结束行`，括号内写 `(现: ...)` 摘要
2. 新建文件不标行号，但列出函数/类/常量名
3. 行号必须用 Read 确认，不得猜测
4. 修改超 30 行，拆成多个子区域

**粒度：** 每条 todo 2-5 分钟可完成，能独立验证，一个功能单元一条。

**格式模板：**

````markdown
# 版本 N: 简短中文需求名（≤8 字）

> 创建时间: YYYY-MM-DD HH:mm:ss
> 最后更新: YYYY-MM-DD HH:mm:ss
> 状态: 待执行

## 需求描述
原始需求。

## 技术方案
选定方案及理由（或单方案思路）。

## 文件变更清单

**新建:** `path` — 职责
**修改:** `path:行号` — 改什么
**依赖:** 包名 — 用途

## Todolist

- [ ] 1. 任务标题

  **Files:**
  - Create: `path`
  - Modify: `path:42-58`

  ```
  change details:
  新建: path
  - 导出的函数/常量/类型
  修改: path:42-58 (现: 当前内容摘要, 改后: 目标状态)
  - 具体改动点
  ```
----------------------
- [ ] 2. ...

## 总结

共 N 条，涉及 X 文件（新建 Y，修改 Z），约 M 行。
1. 任务标题 — 做什么、涉及文件、关键函数、改成什么样
2. ...

## 改动前后对比

直观展示宏观效果变化（有数据贴数据，有请求贴请求响应）。

改动前:
（原始现状）

改动后:
（变化后样子）

规则：基于实际代码、数据优先、不重复 todolist。

## 变更记录
- YYYY-MM-DD HH:mm:ss 初始创建
````

### 5. 等待确认

**AskUserQuestion：**
- question: "以上计划是否确认？"
- options: `确认写入` / `修改意见`

选「修改意见」或 Other → 按反馈调整后重出草案。

### 6. 写入文件

- 目录已存在（`/xz-discuss` 创建过） → 直接写 `N-PLAN.md`
- 不存在 → 从需求提取中文目录名（≤20 字），新建 `.xz_planning/phases/N.中文名/`

写入 `N-PLAN.md`，然后：

```bash
python3 <脚本绝对路径> update-state
```

### 7. 输出结果

```
计划已写入:
  .xz_planning/phases/N.xxx/N-PLAN.md
```

**AskUserQuestion：**
- question: "接下来要做什么？"
- options: `/xz-exec N` / `/xz-update-plan N` / `/xz-ref N`

用户选定后执行对应 skill。

---

## 关键规则

1. **先输出后写** — 草案先展示，用户确认后才落盘
2. **change details 必带行号和当前摘要** — 新建文件列函数/类/常量
3. **零上下文假设** — 写得像给完全不了解项目的人看
4. **原子粒度** — 每条 todo 2-5 分钟，独立可验证
5. **方案来自代码分析** — 禁止凭空凑数；多方案等用户选
6. **调研先行** — 不读代码不规划
7. **多子系统拆分** — 独立子系统各自一个版本
8. **遵循现有模式，DRY/YAGNI** — 不为假想未来预留
9. **版本已存在拒绝** — 导向 `/xz-update-plan N`
